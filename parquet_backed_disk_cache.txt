Parquet-backed NFS cache for DataContainer objects
==================================================

Goal
----

Cache DataContainer payloads whose data is a Polars DataFrame. Cold loads can
come from slow sources such as Oracle or local parquet files. Warm loads read a
Parquet cache file from an NFS-mounted shared directory with Polars.

Current implementation
----------------------

- NFSCache exposes two decorators:
  - @nfscache.parquet for file or function sources.
  - @nfscache.sql for Oracle SQL sources.
- Data is persisted with pyarrow.parquet.ParquetWriter and read with
  polars.read_parquet.
- Cache writes use unique *.part files and os.replace for final replacement.
- Per-key locking is a mkdir-based read/write lock:
  - warm readers create per-reader lock tokens and can overlap
  - writers and invalidations create writer intent, block new readers, and wait
    for active readers to finish
  - reader tokens and writer intent include lock.json metadata with hostname,
    PID, UUID, created_at, and last_seen
  - held locks update last_seen with a heartbeat while the process is alive
  - stale reader/writer locks are broken after stale_lock_seconds
  - lock cleanup removes empty reader/writer directories after release
- Metadata sidecars are authoritative commit markers. They store source_key,
  source_version, parquet byte size, parquet SHA-256, row count, column count,
  schema hash, writer version, created_at, and normalized source_sql for
  SQL-backed entries.
- Readers reject cache entries with missing, stale, unsupported, or corrupt
  metadata, and validate parquet size/checksum/row count/schema hash before a
  warm hit.
- File sources use SHA-256 content hashes by default.
- SQL sources use normalized SQL for cache keys and an Oracle version token:
  COUNT(*) plus MAX(ORA_ROWSCN) for the detected FROM table.
- Cold loads re-read the source version before and after loading and retry if
  the source changed during the read.

Behavior checks
---------------

- main.py clears __cache__ at startup, then demonstrates cold load, warm hit,
  source regeneration, stale invalidation, reload, and warm hit.
- swarm_file.py exercises multi-process cache use and source regeneration.
- tests/ contains focused unittest coverage for authoritative metadata,
  corrupted cache recovery, normalized SQL metadata, overlapping warm readers,
  and writer-preference locking.

Useful commands:

  uv run --no-cache --no-sync python -m unittest discover -s tests
  uv run --no-cache --no-sync python -m compileall -q disk_cache database tests main.py swarm_file.py
  uv run --no-cache --no-sync python -m main
  uv run --no-cache --no-sync python -m swarm_file

What is left to be enterprise-ready
-----------------------------------

This is a working prototype, not an enterprise-ready cache yet. The remaining
work is mostly operational hardening, consistency guarantees, and automated
failure coverage.

1. Validate on the real NFS mount

The design relies on mkdir for lock tokens and os.replace for atomic
replacement. Those are the right primitives, but behavior depends on NFS
version, mount options, attribute caching, server implementation, and multi-host
timing. Prove same-key and different-key behavior on the actual deployment NFS,
not only on local disk.

2. Strengthen the Oracle consistency model

SQL invalidation now uses COUNT(*) and MAX(ORA_ROWSCN), but enterprise use needs
a documented snapshot policy. Long reads should be tied to a consistent Oracle
SCN, for example with flashback query or an equivalent transaction strategy.
Also decide whether tables require ROWDEPENDENCIES, how deletes/truncates/DDL
are detected, and how joins or multi-table SQL should produce a version token.

3. Add observability

Replace print-based status with structured logs and metrics:

- cache hit, miss, reload, and invalidation counts
- source version and cache key
- reader/writer lock wait time
- cold load duration
- parquet write/read duration
- stale lock recovery count
- corruption and retry failures

4. Broaden automated failure tests

The current unit tests and swarm run are useful, but enterprise readiness needs
repeatable tests for:

- crashed reader while holding a read token
- crashed writer while holding writer intent
- crashed writer after writing parquet but before metadata
- more corrupted parquet and metadata variants
- source changes during a cold DB read
- concurrent same-key clients across processes
- concurrent different-key clients across processes
- NFS-mounted integration runs across multiple hosts or containers

5. Add operational controls

Add cache retention, quota management, cleanup of old *.part files, cache
version migration, configurable compression, permissions checks, and clear
runbook steps for clearing bad keys without deleting the whole cache.

6. Harden configuration and security

Move secrets out of defaults for production, define required environment
variables, restrict cache directory permissions, and audit SQL/table parsing.
The current Oracle defaults are for local development only.

Recommended next steps
----------------------

1. Validate mkdir, per-reader tokens, writer intent, stale-lock recovery, and
   os.replace on the target NFS mount.
2. Upgrade Oracle reads to a documented consistent-SCN strategy.
3. Add structured logging and metrics.
4. Broaden crash/corruption/source-change tests.
5. Add operational controls for retention, quotas, cleanup, and bad-key runbooks.
