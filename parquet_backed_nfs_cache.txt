Parquet-backed NFS cache for DataContainer objects
==================================================

Goal
----

Cache DataContainer payloads whose data is a Polars DataFrame. Cold loads can
come from slow sources such as Oracle or local parquet files. Warm loads read a
Parquet cache file from an NFS-mounted shared directory with Polars.

Current foundation
------------------

- DBCache exposes two decorators:
  - @dbcache.data_container_cache for file or function sources.
  - @dbcache.sql_container_cache for Oracle SQL sources.
- Data is persisted through pyarrow.parquet.ParquetWriter and read with
  polars.read_parquet.
- Cache writes use unique *.part files and os.replace for final replacement.
- Each cache key has a mkdir-based lock directory to serialize writers.
- Metadata sidecars store source_version for invalidation.
- File sources use SHA-256 content hashes by default.
- SQL sources use normalized SQL for cache keys and an Oracle version token:
  COUNT(*) plus MAX(ORA_ROWSCN) for the detected FROM table.
- Cold loads re-read the source version before and after loading and retry if
  the source changed during the read.
- main.py is the local smoke test. swarm.py exercises multi-process cache use.

What is left to be enterprise-ready
-----------------------------------

This is a working prototype, not an enterprise-ready cache yet. The remaining
work is mostly operational hardening, consistency guarantees, and automated
failure coverage.

1. Validate on the real NFS mount

The design relies on mkdir for locks and os.replace for atomic replacement.
Those are the right primitives, but behavior depends on NFS version, mount
options, attribute caching, server implementation, and multi-host timing. Prove
same-key and different-key behavior on the actual deployment NFS, not only on
local disk.

2. Add stale lock detection and recovery

Current lock directories can survive a crashed writer forever. Add lock
metadata with hostname, PID, start time, UUID, and heartbeat or timestamp. Define
when a lock may be broken, how competing clients break it safely, and how that
event is logged.

3. Strengthen the Oracle consistency model

SQL invalidation now uses COUNT(*) and MAX(ORA_ROWSCN), but enterprise use needs
a documented snapshot policy. Long reads should be tied to a consistent Oracle
SCN, for example with flashback query or an equivalent transaction strategy.
Also decide whether tables require ROWDEPENDENCIES, how deletes/truncates/DDL
are detected, and how joins or multi-table SQL should produce a version token.

4. Make metadata authoritative

Metadata currently records source_version only. Add enough information to
validate the cache entry: parquet byte size, checksum, row count, schema hash,
writer version, created time, and source key. Define the commit protocol clearly
so readers can distinguish complete entries, stale entries, corrupt files, and
interrupted writes.

5. Improve read concurrency

Reads and writes for the same key are currently serialized through one lock.
That is safe but can limit throughput. Evaluate lock-free reads with stricter
metadata ordering, or a read/write lock that allows many readers while blocking
only writers and invalidations.

6. Add observability

Replace print-based status with structured logs and metrics:

- cache hit, miss, reload, and invalidation counts
- source version and cache key
- lock wait time
- cold load duration
- parquet write/read duration
- stale lock recovery count
- corruption and retry failures

7. Build automated failure tests

The swarm run is useful, but enterprise readiness needs repeatable tests for:

- crashed writer while holding a lock
- crashed writer after writing parquet but before metadata
- corrupted parquet file
- corrupted metadata file
- source changes during a cold DB read
- concurrent same-key clients
- concurrent different-key clients
- NFS-mounted integration runs across multiple hosts or containers

8. Add operational controls

Add cache retention, quota management, cleanup of old *.part files, cache
version migration, configurable compression, permissions checks, and clear runbook
steps for clearing bad keys without deleting the whole cache.

9. Harden configuration and security

Move secrets out of defaults for production, define required environment
variables, restrict cache directory permissions, and audit SQL/table parsing.
The current Oracle defaults are for local development only.

Recommended next steps
----------------------

1. Implement lock metadata plus stale-lock recovery.
2. Add pytest coverage for crash, corruption, and source-change cases.
3. Validate mkdir and os.replace semantics on the target NFS mount.
4. Upgrade Oracle reads to a documented consistent-SCN strategy.
5. Add structured logging and metrics before broader rollout.
